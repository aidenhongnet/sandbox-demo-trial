## Your task

Reach a specific product screen and capture its state for downstream verification.

## Instructions

1. **Navigate** to the target screen described in the plan. Use the navigation_hints as high-level guidance — adapt if the UI differs slightly from what the hints describe. Click elements by their visible labels, wait for pages to load, scroll if needed.

2. **Verify arrival** before capturing anything. Confirm you are on the correct screen by checking that the URL, page title, or visible content matches the plan's goal. If something looks wrong, try to recover by navigating back and retrying. Do not capture a wrong screen.

3. **Take a screenshot** once you are confident you are on the correct screen. Save it to:
   {screenshot_path}

4. **Describe every visible UI element** in thorough detail. For each element you can see, note:
   - Element type (button, link, input field, dropdown, toggle, tab, icon, label, panel, card, table, etc.)
   - Exact visible text or label
   - Current state: enabled/disabled, selected/unselected, expanded/collapsed, checked/unchecked, active/inactive
   - Position context: which section or panel it belongs to, what it is near
   - For dropdowns or selects: the currently selected value if visible
   - For input fields: any placeholder text or current value
   - For toggles/checkboxes: on/off state
   - For tables: column headers and a summary of visible rows
   - For icons: describe their appearance and what they likely represent

5. **For behavior claims** listed in what_to_capture: if the plan asks you to verify that an action produces a specific result, perform the action, observe the result, and describe both the before and after states clearly. Take additional screenshots if the UI changes.

6. **Write the description** to this file using the Write tool:
   {description_path}

   The description should be a structured, plain-text inventory of everything visible on the screen. Organize by visual sections (e.g., sidebar, header, main content area, modals). Be exhaustive — a downstream verifier will use this text to confirm or deny specific documentation claims without seeing the screenshot.

## Important rules

- If you CANNOT reach the target screen (navigation fails, element not found after retries, unexpected error), do NOT write any files. Simply report what went wrong. The orchestrator detects missing files as a failure signal.
- Do not fabricate or hallucinate UI content. Only describe what you actually see in the browser.
- Use the screenshot tool to see the current state of the page before describing it.
- If the page has scrollable content, scroll and capture additional screenshots to ensure you describe ALL visible elements, not just what's above the fold.
