## Instructions

1. **Navigate** to the target screen. Use the navigation_hints as high-level guidance — adapt if the UI differs. Click elements by their visible labels, wait for loads, scroll if needed.

2. **Verify arrival** before capturing. Confirm the URL, page title, or visible content matches the plan's goal. If something looks wrong, recover by navigating back and retrying. Do not capture a wrong screen.

3. **Record a navigation trace as you go.** Keep an ordered list of every action you take. After arriving, write the trace as JSON to:
   {trace_path}

   Use exactly this shape:
   {"steps": [{"step": 1, "action": "navigate|click|type|select", "target_label": "the visible label/control you acted on", "resulting_url": "URL after the action", "resulting_title": "page/section title after the action"}], "final_url": "...", "final_title": "..."}

   Record what you actually clicked and where it actually led — not what you expected. This trace is how navigation-path claims get verified.

4. **Take a screenshot** once you are confident you are on the correct screen. Save it to:
   {screenshot_path}

5. **Describe every visible UI element** in thorough, neutral detail — an exhaustive inventory of the region, not a check of any particular claim. For each element note:
   - Element type (button, link, input, dropdown, toggle, tab, icon, label, panel, card, table, etc.)
   - Exact visible text or label
   - Current state: enabled/disabled, selected, expanded/collapsed, checked, active
   - Position context: which section/panel it belongs to, what it is near
   - For dropdowns/selects: the currently selected value if visible
   - For inputs: placeholder or current value
   - For toggles/checkboxes: on/off state
   - For tables: column headers and a summary of visible rows
   - For icons: appearance and what they likely represent
   - Explicitly note what is NOT present if a region is empty or a common control is absent

6. **For action-dependent state** named in what_to_capture: if reaching the state requires performing an action, perform it, then describe the before and after states. Take additional screenshots if the UI changes.

7. **Write the description** to this file using the Write tool:
   {description_path}

   Organize by visual sections (sidebar, header, main content, modals). Be exhaustive — the downstream verifier sees only this text (and possibly the screenshot), never the live page.

## Important rules

- If you CANNOT reach the target screen (navigation fails, element not found after retries, unexpected error), do NOT write the screenshot or description files. Report what went wrong. The orchestrator treats missing files as an honest failure signal.
- Do not fabricate or hallucinate UI content. Only describe what you actually see.
- Use the screenshot tool to observe the current state before describing it.
- If content is scrollable, scroll and capture additional screenshots so you describe ALL visible elements.
